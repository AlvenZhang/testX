# AI 自动化测试平台 - 使用流程介绍

## 一、平台概述

本平台是一个**多项目AI驱动的自动化测试平台**，实现从"需求 + 代码变更"到"测试执行 + 回归测试"的全流程自动化。

**核心特性：**
- 🤖 AI 全流程赋能：需求分析 → 用例生成 → 代码生成 → 代码修复
- 📱 移动端测试支持：Android + iOS 真机/模拟器
- 🐳 Docker 沙箱隔离：Web/API 测试环境隔离
- 👥 多项目权限管理：灵活的角色权限控制
- 📊 可视化报告：数据图表 + 多格式导出

---

## 二、快速开始

### 2.1 环境启动

```bash
cd ai-test-platform
docker-compose up -d
```

启动后访问：
- 前端：http://localhost:3000
- 后端 API：http://localhost:8000
- API 文档：http://localhost:8000/docs

### 2.2 首次登录

1. 访问前端页面，点击"注册"创建账号
2. 使用注册的邮箱和密码登录
3. 进入主界面

## 三、完整使用流程

### 步骤 1：创建项目

1. 点击左侧菜单"项目管理"
2. 点击"新建项目"按钮
3. 填写项目信息：
   - 项目名称：如"电商平台测试"
   - 项目描述：简要描述项目
   - Git 仓库地址（可选）：绑定代码仓库
   - 项目配置：选择测试类型（web/android/ios/api）
4. 点击"创建"完成

**项目配置示例：**
```json
{
  "test_types": ["web", "api"],
  "default_framework": "pytest"
}
```

---

### 步骤 2：添加项目成员（可选）

1. 进入项目详情页
2. 点击"成员管理"标签
3. 点击"添加成员"
4. 选择用户并分配角色：
   - `viewer`：查看者（只读）
   - `editor`：编辑者（可编辑需求和用例）
   - `executor`：执行者（可执行测试）
   - `admin`：管理员（所有权限）

---

### 步骤 3：录入需求

1. 点击左侧菜单"需求管理"
2. 选择目标项目
3. 点击"新建需求"
4. 填写需求详情：
   - **标题**：简洁明了的需求标题
   - **描述**：详细描述需求，包括功能点、业务流程、约束条件等
   - **优先级**：低/中/高/紧急
   - **附件**（可选）：上传需求文档（PDF/Word/Markdown）
5. 点击"创建"

**需求描述示例：**
```
用户登录功能：
1. 输入正确的用户名和密码，点击登录后跳转到首页
2. 输入错误的用户名或密码，显示错误提示"用户名或密码错误"
3. 连续3次登录失败，显示验证码
4. 登录成功后记录登录时间和IP
```

---

### 步骤 4：AI 生成测试用例

1. 在需求列表中找到刚创建的需求
2. 点击"AI 生成"按钮 🚀
3. 等待 AI 分析（约 10-30 秒）
4. 查看生成结果：
   - **测试要点**：AI 识别的核心测试点
   - **测试用例**：结构化的用例表格，包含：
     - 用例 ID
     - 标题
     - 测试步骤
     - 预期结果
     - 优先级
5. （可选）编辑用例：可以修改、删除或新增用例
6. 点击"确认"进入代码生成

**生成的测试用例示例：**

| 用例ID | 标题 | 步骤 | 预期结果 | 优先级 |
|--------|------|------|---------|--------|
| TC001 | 正确用户名密码登录 | 1. 输入用户名admin<br>2. 输入密码123456<br>3. 点击登录 | 跳转到首页 | high |
| TC002 | 错误密码登录 | 1. 输入用户名admin<br>2. 输入密码wrong<br>3. 点击登录 | 显示"用户名或密码错误" | high |

---

### 步骤 5：AI 生成测试代码

1. 确认测试用例后，自动进入代码生成阶段
2. 选择测试框架和类型：
   - 框架：pytest / TestNG / Playwright
   - 类型：API / Web / Mobile
3. 点击"生成代码"
4. 等待 AI 生成（约 15-45 秒）
5. 查看生成的代码：
   - 在 Monaco Editor 中查看和编辑
   - 支持语法高亮、代码折叠
   - 可以手动调整代码
6. 点击"保存"

**生成的 pytest 代码示例：**
```python
import pytest
import httpx

BASE_URL = "http://localhost:8000"

@pytest.mark.asyncio
async def test_TC001_correct_login():
    """TC001: 正确用户名密码登录"""
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"{BASE_URL}/auth/login",
            json={"email": "admin@example.com", "password": "123456"}
        )
        assert response.status_code == 200
        assert "access_token" in response.json()

@pytest.mark.asyncio
async def test_TC002_wrong_password():
    """TC002: 错误密码登录"""
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"{BASE_URL}/auth/login",
            json={"email": "admin@example.com", "password": "wrong"}
        )
        assert response.status_code == 401
```

---

### 步骤 6：执行测试

#### 6.1 Web/API 测试（Docker 沙箱）

1. 进入"测试代码"页面
2. 选择要执行的测试代码
3. 点击"执行"按钮 ▶️
4. 实时查看执行日志：
   - WebSocket 实时推送日志
   - 显示执行进度
   - 查看控制台输出
5. 等待执行完成
6. 查看执行结果：
   - 状态：成功/失败
   - 执行时长
   - 详细日志

#### 6.2 移动端测试

1. 确保设备已连接（Android/iOS）
2. 进入"设备管理"页面
3. 点击"发现设备"扫描可用设备
4. 选择目标设备
5. 进入"测试代码"页面
6. 选择移动端测试代码
7. 点击"执行"
8. 实时查看：
   - 设备屏幕截图
   - 执行日志
   - Appium 输出

---

### 步骤 7：查看测试报告

1. 测试执行完成后，自动生成报告
2. 进入"测试报告"页面
3. 选择对应的测试运行
4. 查看报告详情：
   - **数据概览**：
     - 总用例数
     - 通过数
     - 失败数
     - 通过率饼图
     - 执行时长
   - **趋势图表**：测试趋势折线图
   - **失败详情**：每个失败用例的错误信息
   - **日志内容**：完整的执行日志
   - **截图**（UI测试）：操作步骤截图
5. 导出报告：
   - 点击"导出 CSV"
   - 或"导出 HTML"
   - 下载报告文件

---

### 步骤 8：需求变更与影响分析

#### 8.1 修改需求

1. 在需求列表中找到需要修改的需求
2. 点击"编辑"
3. 修改需求内容
4. 保存后自动保存版本历史

#### 8.2 AI 影响分析

1. 点击"AI 影响分析"按钮
2. 等待 AI 分析（约 20-40 秒）
3. 查看影响分析报告：
   - **受影响的历史需求**：哪些旧需求可能受影响
   - **受影响的测试代码**：哪些旧测试代码可能失效
   - **影响等级**：高/中/低
   - **分析原因**：详细的影响说明
4. 根据分析结果决定是否需要修复旧代码

#### 8.3 自动修复旧测试代码

1. 在影响分析结果中，点击"修复代码"
2. AI 自动读取：
   - 变更后的新需求
   - 受影响的旧需求
   - 失效的旧测试代码
3. 生成修复后的代码
4. 查看代码对比（Diff Viewer）：
   - 红色：删除的代码
   - 绿色：新增的代码
5. 确认无误后，点击"覆盖旧代码"

---

### 步骤 9：回归测试

1. 进入"测试计划"页面
2. 创建回归测试计划
3. 选择需要回归的测试代码
4. 或使用 AI 建议的回归测试列表
5. 点击"批量执行"
6. 等待所有测试完成
7. 查看回归测试报告

## 四、移动端测试专项

### 4.1 Android 设备设置

**前置条件：**
- Android 真机或模拟器
- 已开启 USB 调试
- 电脑已安装 ADB

**连接步骤：**
```bash
# 1. USB 连接设备
adb devices

# 2. 开启网络 ADB（可选）
adb tcpip 5555

# 3. 网络连接（WiFi）
adb connect 192.168.1.100:5555
```

**平台操作：**
1. 进入"设备管理"
2. 点击"发现设备" → 选择 "Android"
3. 系统自动扫描并添加设备
4. 设备显示为"在线"状态
5. 可以点击"截图"预览设备屏幕

### 4.2 iOS 设备设置

**前置条件：**
- macOS 电脑
- iOS 真机或模拟器
- Xcode 已安装
- Appium Desktop 已安装

**连接步骤：**
```bash
# 列出可用的模拟器
xcrun simctl list devices available

# 启动模拟器
xcrun simctl boot <DEVICE_UUID>
```

**平台操作：**
1. 进入"设备管理"
2. 点击"发现设备" → 选择 "iOS"
3. 系统自动发现模拟器
4. 选择设备并连接

## 五、常见问题

### Q1: AI 生成的代码无法运行怎么办？

A:
1. 先查看错误日志，了解具体错误
2. 在代码编辑器中手动修改
3. 或点击"重新生成"，调整提示词
4. 或使用"代码修复"功能

### Q2: 测试执行超时怎么办？

A:
1. 检查 Docker 沙箱是否正常运行
2. 查看测试代码是否有死循环
3. 增加超时时间配置
4. 拆分测试用例为更小的单元

### Q3: 移动端设备连接失败？

A:
1. Android: 检查 USB 调试是否开启
2. Android: 确认 ADB 能识别设备 `adb devices`
3. iOS: 确认在 macOS 上运行
4. iOS: 检查 Xcode 和模拟器状态

### Q4: 如何备份数据？

A:
1. 定期导出数据库
2. 使用 Docker 卷备份
3. 重要报告导出保存

## 六、最佳实践

### 6.1 需求描述技巧

**好的需求描述应该包含：**
- ✅ 明确的功能点
- ✅ 正常流程和异常流程
- ✅ 业务规则和约束条件
- ✅ 输入输出示例

**示例：**
```
好：用户登录时，输入正确的邮箱和密码跳转到首页，
输入错误时显示"邮箱或密码错误"，连续3次失败需要验证码。

不好：做一个登录页面。
```

### 6.2 测试用例维护

- 定期审查和更新测试用例
- 及时删除过时的用例
- 保持用例的独立性和可重复性
- 使用有意义的用例名称

### 6.3 团队协作建议

1. **分工明确**：
   - 产品经理：录入需求
   - 测试工程师：审查和调整用例
   - 开发工程师：协助调试测试代码

2. **流程规范**：
   - 需求变更前先做影响分析
   - 定期执行回归测试
   - 保存重要的测试报告

3. **权限设置**：
   - 产品经理：editor
   - 测试工程师：executor
   - 技术负责人：admin
